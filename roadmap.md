# Lattice Roadmap

What's done, what's next, what's deferred. Sequenced for maximum utility as a standalone library management suite. Updated as of v4.8.1.

---

## Phase 1: Foundation & Extraction (Data Layer)
- [x] **Unified Tag Reader**: `get_all_tags` returning `TagBundle` from a single `MutagenFile()` open.
- [x] **Universal Audio Support**: Extract metadata across MP3, FLAC, OGG, Opus, M4A, WAV, WMA, AAC.
- [x] **Rating Normalization**: Convert POPM, TXXX, and Vorbis comments (0-100, 0-255 scales) into a standard 0-5 float.
- [x] **Cover Art Priority Extraction**: Extract embedded art prioritizing FLAC > Opus > M4A > MP3.
- [x] **Front Cover Preference**: Prefer ID3 APIC/FLAC type 3 over generic embedded images.
- [x] **Modern Architecture**: Migrate from monolithic script to a modular `src/lattice` package via Hatch (`pyproject.toml`).
- [x] **Configurable Layout**: Support non-standard directory structures via pattern argument instead of strict `ARTIST/ALBUM/Track.ext`.
- [x] **Fix Close button trying to change music root location** - This should just exit.

## Phase 2: Integrity & Auditing (Validation Layer)
- [x] **Parallel FLAC Verification**: Spawn parallel workers to run `flac -t` or `ffmpeg` to detect corruption.
- [x] **Unified Decode Scanner**: Run parallel FFmpeg decodes for MP3 and Opus formats to detect decode errors.
- [x] **Metadata Audit**: Report audio files missing critical tags (title, artist, track number, genre).
- [x] **Duplicate Detection**: Detect the same artist + album combination appearing across multiple directories or formats.
- [x] **Missing Art Reporting**: Distinguish between directories with absolutely no art and those with only embedded art.
- [x] **Case-Insensitive Cover Detection**: Detect `cover.jpg`, `Folder.png`, etc. without case sensitivity issues.
- [x] **WAV/WMA Integrity**: Extend the unified decode scanner to cover remaining legacy formats.
- [x] **Art Quality Audit**: Report extracted/folder covers below a resolution threshold (e.g., < 500x500).
- [x] **Bitrate Floor Audit**: Flag audio files falling below a configurable bitrate floor (e.g., < 192 kbps).
- [x] **ReplayGain Audit**: Report per-album ReplayGain coverage (missing / partial / no album gain / OK), format-aware so Opus R128 gain counts as tagged. (v4.8.0)

## Phase 3: Reporting & Generation (Output Layer)
- [x] **Library Tree Generation**: Build beautiful ASCII/Unicode trees displaying artists, albums, tracks, and ratings.
- [x] **Genre Appending**: Optional `--genres` flag to inline genre tags into the library tree.
- [x] **AI-Readable Export**: Generate a token-efficient, pipe-delimited library dump designed for LLM prompts.
- [x] **Genre Wings**: Generate separate library `.txt` files for every detected genre in the collection.
- [x] **AI Wings**: Generate separate, token-efficient AI library files for each genre (Artist | Album | Genre | Location).
- [x] **Library Statistics**: Generate reports on formats, bitrates, top artists, top genres, and rating distributions.
- [x] **Playlist Export (.m3u)**: Generate standard `.m3u` playlists from library tree filters (e.g., 5-star only, specific genre).
- [x] **Smart Playlists**: Dynamic rule-based `.m3u` generation (e.g., "rating >= 4 AND genre == 'Jazz'").
- [x] **Rating Distribution per Genre**: Cross-tabulate rating spread specific to each genre inside `--stats`.

## Phase 4: Interface & Experience (UX Layer)
- [x] **Full-Screen Curses TUI**: Arrow-key navigation, color-coded section groups, and styled Unicode box drawing.
- [x] **Immersive TUI Operations**: In-menu progress bars and integrated text pagers that never drop the user back to a raw terminal shell.
- [x] **Graceful Fallbacks**: Text-based menu and typed input for environments without `curses` support.
- [x] **Persistent Configuration**: Save library root to `~/.config/lattice/config.json` on first run to eliminate redundant prompts.
- [x] **Submenus**: Nested library tree/export modes inside the TUI.
- [x] **Standalone Binary**: Support compiling a self-contained, native binary using PyInstaller.
- [x] **CLI Flag Parity**: Extensive command-line arguments mapping 1:1 with TUI capabilities (including `--dry-run` and `--version`).
- [ ] **Progress Persistence**: Resume interrupted large-scale integrity scans (FLAC/MP3) from where they left off without restarting.
- [x] **Color Output in CLI**: Status summaries are colorized (green all-clear, yellow suspect, red corrupt). Gated on an interactive TTY: off in the TUI, off when piped or redirected, off under `NO_COLOR`, so reports and pipes stay clean. (v4.6.0)
- [x] **Multi-Root Scanning**: `--root` is repeatable; several libraries scan together in one pass and de-dupe a repeated path. Cross-library duplicate detection groups an album that lives in two libraries, with each entry prefixed by its root's basename. An optional `library_roots` array in the JSON config supplies default roots; the first-run prompt still saves only the single `library_root`. (v4.6.0)
- [x] **Configurable Path Layout**: A `layout` config key (and `--layout`) sets the pattern Lattice uses to recover artist/album/genre from a path, so a genre-first tree (`{genre}/{artist}/{album}`) is fully supported; genre now falls back to the path like artist/album already did. Default stays `{artist}/{album}`. Pairs with the `genre_foldermap.py` companion script that builds such a tree. (v4.7.0)

## Found Bugs

- [x] **FOUND BUG (2026-06-10): TUI fallback menu desync.** The no-curses typed-input menu's hand-maintained key map went stale as modes were added: "6" was labelled "Test WAV files" but dispatched Extract cover art (a library-writing mode), "7"–"9" were each shifted one mode group over, "10"–"13" were rejected as invalid, and the WAV/WMA tests, art-quality audit, bitrate audit, and ReplayGain audit were unreachable from the fallback. The curses menu was unaffected. **Fixed (v4.8.1):** the fallback listing and its key map are now both generated from the same sections the arrow-key menu renders (`_build_fallback` in `tui.py`), so they cannot drift again; `tests/test_tui.py` pins the alignment.
- [x] **FOUND BUG (2026-05-25): `cleaner.py` apostrophe normalization gap.** A fragmented album survived the initial consolidation pass: JPEGMAFIA's *I LAY DOWN MY LIFE FOR YOU (Director's Cut)* was split across two sibling folders, `I LAY DOWN MY LIFE FOR YOU (DIRECTOR's CUT)/` (12 tracks, mp3+opus) and `I LAY DOWN MY LIFE FOR YOU (DIRECTORS CUT)/` (4 unique tracks, mp3). `cleaner.py` did not merge them because its normalization folded curly to straight quotes but did not equate apostrophe-present vs apostrophe-absent: `director's cut` and `directors cut` hashed to different keys. **Fixed (v4.6.0):** `normalize_name` now strips apostrophes and collapses whitespace after the NFKC/dash/quote/case folding, so `X's` and `Xs` collapse to the same key. Confirmed on the real library (it also caught a Damian "Jr. Gong" Marley quote variant).

## Audit 2026-07-01: TUI/UX + companion scripts (v4.8.1 baseline)

Full audit of `src/lattice/tui.py` plus all seven companion scripts, run 2026-07-01 against v4.8.1. Every finding below was verified against the source; the load-bearing ones were reproduced empirically (cleaner/genre_tidy against temp trees and the fixture library; replaygain against the installed `rsgain`). All 306 tests passed before the audit; every bug lives in an untested corner, so **each fix lands with the test that would have caught it**. Line numbers are as of v4.8.1 and will drift; the anchors (function names, quoted strings) are the durable reference.

Housekeeping rules for working this list: companion-script fixes get a dated `patchnotes.md` entry with a bumped script version in its docstring header where one exists (no package version bump); `tui.py`/`utils.py` fixes are package changes and bump `config.VERSION` + `pyproject.toml` + `spec.md` §1 together. Fix order: H1–H7 first (data corruption/loss on the real library), then mediums per script, then lows/UX. Tick boxes as items ship.

### High severity

- [ ] **H1 (cleaner.py): `_FEAT_RE` matches `ft` inside words, corrupting artist tags non-idempotently.** `_FEAT_RE = re.compile(r"\s*(?:feat\.?|ft\.?|featuring)\s+(.*)$", re.I)` (cleaner.py:193): the leading `\s*` is zero-width, so the `ft` at the end of "Left", "Swift", "Croft" matches. Reproduced: `canon_track_artist("Left Boy", "Left Boy")` returns `"Left Boy feat. Boy"`; a second `--normalize-tags` authority pass produces `"Left Boy feat. Boy feat. Boy"` (unbounded growth). Any merged/renamed artist folder whose name contains a mid-word `ft`/`feat` gets clean tags rewritten wrong.
  **Fix:** require a real token boundary before the feat marker: `re.compile(r"(?:^|[\s(])(?:feat\.?|ft\.?|featuring)\s+(.*)$", re.I)` or equivalently add `(?<!\w)` before the alternation. While here, close the sibling paren bug (L7 below) in the same change: when the matched region began with `(` and the guest capture ends with an unmatched `)`, strip that trailing `)` from `m.group(1)` (`canon_track_artist("A (feat. B)", "A")` currently returns `"A feat. B)"`).
  **Tests (tests/test_cleaner.py, `CanonTrackArtistTests`):** `("Left Boy", "Left Boy") == "Left Boy"`; `("Left Lane Cruiser", "Left Lane Cruiser") == "Left Lane Cruiser"`; `("A (feat. B)", "A") == "A feat. B"`; idempotency: `canon_track_artist(canon_track_artist(x, c), c) == canon_track_artist(x, c)` for a real feat case.

- [ ] **H2 (cleaner.py): `.wma` missing from `AUDIO_EXT`, so a differing-size `.wma` collision is deleted.** `AUDIO_EXT` (cleaner.py:82-94) lacks `.wma` while `TAGGABLE_EXT` (cleaner.py:80) and the package's canonical `AUDIO_EXTENSIONS` (src/lattice/config.py:29) both include it. Reproduced: merging `album/01.wma` (differing size) into `Album/01.wma` unlinks the source via the DROP NON-AUDIO branch (cleaner.py:449-455) instead of keeping a `.from-fragment` copy. Direct violation of the documented "audio is never overwritten or deleted" guarantee. Same gap makes `--normalize-filenames` skip `.wma` files.
  **Fix:** add `".wma"` to `AUDIO_EXT`. Make a deliberate call on `".mp4"` (in `TAGGABLE_EXT`, not `AUDIO_EXT`; defensible to exclude since `.mp4` is ambiguous with video, but document the choice in a comment). Audit `AUDIO_EXT` against `config.AUDIO_EXTENSIONS` once and note any intentional differences inline so they can't read as accidents.
  **Tests:** a merge fixture with a differing-size `.wma` collision asserting the source survives as `01.from-fragment.wma`; a `--normalize-filenames` case renaming a `.wma` track.

- [ ] **H3 (cleaner.py): dry-run misses the survivor rename because the collision check ignores virtual removals.** `_normalize_folder_name` (cleaner.py:475) gates on `dst.exists()`; in dry-run the merged-away source folder still exists on disk, so the survivor rename in the flagship scenario (canonical `Drive‐By Truckers` U+2010 merging source `Drive-By Truckers`) logs `RETAIN NAME (normalized target exists)` in preview while apply performs the rename. Reproduced: dry stats `{moves:1, rmdirs:1}` vs apply `{moves:1, renamed:1, rmdirs:1}`. Knock-on: the missed rename never records the tag-authority target, so a `--normalize-tags` preview omits restamps apply performs.
  **Fix:** add a helper (e.g. `def _gone(run, p): return p in run.removed`) and change the guard in `_normalize_folder_name` and `_normalize_file_name` (cleaner.py:540) to `if dst.exists() and dst != folder and dst not in run.removed:`. The dry-run `_rename` path must also add the source path to `run.removed` (it becomes nonexistent under the new name) if it doesn't already.
  **Tests:** dry-vs-apply stats equality on the existing `SurvivorRenameTests` fixture (run the same tree twice, once dry, once apply, compare `renamed` and the tag-target set).

- [ ] **H4 (genre_foldermap.py): multi-disc album subfolders misclassified as "organized", poisoning the genre vocabulary.** `classify` (genre_foldermap.py:133-134) calls any depth-3 audio dir organized, and lattice's scanner (`library._scan_album_dirs`) emits one record per audio-bearing directory at any depth. In a *flat* library, `Artist/Album/CD1` yields a depth-3 record, so `"Artist"` enters `allowed_genres` (genre_foldermap.py:196), gating turns on, and every ordinary stray is flagged `UNKNOWN GENRE`: one CD1 folder breaks the advertised flat-to-genre conversion. The disc album itself is never filed (perpetual `NOTE ... left in place`). The same failure routes through the staging inbox: `Unfiltered/Artist/Album/CD1` strips to depth 3, reads as organized, sticks in the inbox forever, and the artist name in the vocabulary can even let an unrelated stray tagged with that artist-name-as-genre pass the gate. Corollary in an *organized* library: legit `Genre/Artist/Album/CD1` dirs are depth 4 and produce perpetual `TOO DEEP` noise.
  **Fix:** teach classify about disc folders. (1) Add `DISC_DIR_RE = re.compile(r"(?i)^(?:cd|disc|disk|dvd|side|vinyl)[\s._-]*\d+$")` near the other constants. (2) In `classify`: if `len(parts) == 3` and `DISC_DIR_RE.match(parts[2])`, return `("album", parts[0], parts[1])` with the move path being the *parent* album dir (`root/parts[0]/parts[1]`), so the album moves as a unit with its discs inside; if `len(parts) == 4` and `DISC_DIR_RE.match(parts[3])`, return `("organized", parts[0], parts[1], parts[2])` instead of toodeep. (3) In `build_plan`, dedupe: CD1 and CD2 both resolve to the same album move; skip when `seen_dst.get(dst) == src` instead of logging a COLLISION. Note the classified path and the record path now differ for disc records; compute genre/vocabulary from the classified result, not the raw record. (4) Vocabulary hygiene: only true depth-3 `organized` records (not disc-derived depth-4 ones, which are already correct) contribute `info[1]` to `allowed_genres`; with the step-2 change this falls out naturally, but assert it in a test. (5) Staging leftovers: after the strip, a stripped path at depth ≥3 that is *not* a disc dir can never legitimately be "organized" relative to the inbox; classify needs to know the strip happened (pass a `staged: bool` derived from the prefix match) and return a new `("staged-toodeep", depth)` kind that `build_plan` reports as an issue (`STAGED TOO DEEP (left in inbox)`), never as organized.
  **Tests (tests/test_genre_foldermap.py):** flat library containing `Artist/Album/CD1+CD2` converts fully (gate stays off, album moves as one unit, one manifest row); organized library with `Genre/Artist/Album/CD1` produces no TOO DEEP issue and no move; `Unfiltered/Artist/Album/CD1` files the album into the taxonomy; `Unfiltered/A/B/C/weird` (non-disc) is flagged, not treated as organized; vocabulary excludes the flat-library artist name.

- [ ] **H5 (genre_tidy.py + retag.py): a slash canonical can never read back compliant, causing a permanent retag loop.** `retag_argv` (genre_tidy.py:133-142) splits `"Emo / Orgcore"` on `/` into two genre args; retag writes them as a multi-value tag; the read-back never equals the map. Reproduced both ways: FLAC gets two `genre` fields and lattice's `TagBundle` reads only the first (`tags.py` `_first_text`), so compliance compares `"emo"` vs `"emo / orgcore"`; MP3's v2.3 save slash-joins `text=["Emo","Orgcore"]` into `"Emo/Orgcore"` (no spaces), which normalizes differently from `"emo / orgcore"`. Every `apply` re-runs retag on the same albums forever: files rewritten, APEv2 stripped, mtimes churned, dry-run never converges. Zero live blast radius today (`artist_genre_defaults.tsv` has slashes only in artist names), but the split is an advertised, tested feature (`test_slash_canonical_splits_into_separate_args`).
  **Fix (recommended):** retire the split. In `retag_argv`, pass the canonical verbatim as a single genre argument, so retag writes one literal genre string `"Emo / Orgcore"` in every container and the round-trip through `norm()` is exact. Update the `retag_argv` docstring (the "'/'-joined canonical is split" sentence) and replace `test_slash_canonical_splits_into_separate_args` with a test asserting the single-arg form. This treats a slash genre as one genre value, which is what DeaDBeeF/foobar display anyway and what the map's compliance check can actually verify.
  **Alternative (only if true multi-genre tags are wanted):** keep the split but (a) extend lattice's `TagBundle` genre read to join multiple Vorbis genre fields with `" / "`, (b) make retag's MP3 path write one pre-joined TCON string instead of a multi-value list, and (c) make `genre_tidy.norm()` normalize separator spacing (`"a/b"` == `"a / b"`). Strictly more moving parts across two codebases; the recommendation stands unless multi-value genre tags become a real requirement.
  **Tests (tests/test_genre_tidy.py):** end-to-end convergence: build a fixture album, apply a slash canonical, re-run compliance on the re-read tags and assert compliant (i.e. a second `apply` plans zero retags) for both a FLAC and an MP3 file.

- [ ] **H6 (tui.py + utils.py): mode execution has no exception boundary and the curses lifecycle can strand the terminal.** Three legs, one cluster. (a) `_run_with_capture` (tui.py:668-670) calls `func(...)` bare and nothing upstream catches non-`KeyboardInterrupt` exceptions, so a mode error (e.g. unwritable output path raising `PermissionError` after a completed scan) escapes as a traceback, loses the results, and exits with the terminal stuck in curses mode because `_TUIPbar.draw` did `curses.initscr()` (utils.py:281) and nothing ever calls `endwin()` on this path. (b) `interactive_menu` sets `utils.IN_TUI = True` unconditionally (tui.py:798) even when `_USE_CURSES` is False (tui.py:65), so the plain-text fallback session hands progress to `_TUIPbar`; if the curses *module* is missing, the `import curses` at utils.py:276 sits before the `try` and raises `ModuleNotFoundError`; if curses exists but stdin isn't a tty, `initscr()` hijacks the screen mid-fallback-session and is never torn down. (c) Ctrl-C in non-integrity modes (audit/stats/playlists don't catch `KeyboardInterrupt`) propagates to `cli.py:381` with no terminal restore.
  **Fix:** (1) In `_run_with_capture`, wrap the `func(...)` call: `except KeyboardInterrupt` pages "[Cancelled]" plus whatever output was captured; `except Exception` captures `traceback.format_exc()` and pages it under an `[Error]` heading; in a `finally`, when `_USE_CURSES`, call `curses.endwin()` guarded by `if not curses.isendwin():` inside its own try/except, then `_reset_terminal()`. (2) Change tui.py:798 to `utils.IN_TUI = _USE_CURSES` so the fallback session never selects `_TUIPbar`; additionally move the `import curses` in `_TUIPbar.draw` inside the `try` (belt and suspenders). (3) Give `_TUIPbar.close()` a real body: `endwin()` behind the same `isendwin()` guard, so well-behaved modes restore the screen as soon as their bar closes. (4) While in utils.py, fix the factually wrong comment at utils.py:279-281 (`curses.wrapper` has already torn the screen down by the time a mode runs; `initscr()` here starts a fresh screen, it does not resume one) and replace the `box_w = 46` magic number and raw `color_pair(1)/(3)` with constants shared with tui.py (`_TUI_BOX_W`, the color-pair ids) imported or mirrored deliberately with a cross-reference comment.
  **Tests (tests/test_tui.py):** `_run_with_capture` with a raising func pages the traceback instead of propagating (monkeypatch `_tui_page` to record); with a `KeyboardInterrupt`-raising func it pages a cancel notice; `utils.IN_TUI` False path: with `IN_TUI` True but a monkeypatched unusable curses, `_make_pbar` returns `_FallbackProgress` (after fix 2 this becomes: fallback sessions never set `IN_TUI`).

- [ ] **H7 (tui.py): curses init failure in `_tui_select` is indistinguishable from Quit; TUI silently exits 0 on capability-poor terminals.** `_run` calls `curses.curs_set(0)` (tui.py:244) and `_init_tui_colors` (tui.py:243) which raise `curses.error` on terminals without those capabilities (`TERM=vt100`, dumb terminals); the outer `except curses.error: return None` (tui.py:260-263) is read at tui.py:815 as "user chose Quit". Net: on such a terminal the first-run prompt works (its widget degrades gracefully), then the menu "opens" and the program instantly exits 0 with no message, even though the `_box_menu` text fallback exists and works.
  **Fix:** (1) Make the cosmetic calls non-fatal: wrap `curses.curs_set(0)` and the color init in individual `try/except curses.error: pass` (a monochrome, visible-cursor TUI beats a dead one). (2) Distinguish real init failure from Quit: catch `curses.error` from `curses.wrapper` itself, log nothing but flip the module-level `_USE_CURSES` to False and return a sentinel (e.g. the string `"fallback"`) that the caller loops on, so the very next iteration renders the text fallback menu. `_tui_prompt_str`, `_tui_pause`, `_tui_page` already degrade; `_tui_select` is the one widget mapping failure to a user action.
  **Tests:** monkeypatch `curses.wrapper` to raise `curses.error` and assert `interactive_menu` falls back to the text menu instead of returning 0; assert `curs_set` raising inside `_run` does not abort selection.

### Medium severity

- [ ] **M1 (apestrip.py): `--keep-metadata` on a v1-only MP3 clobbers the ID3v1 tag.** `ID3(path)` raises `ID3NoHeaderError` for a file with only ID3v1 (mutagen's `ID3` never reads v1), so `id3 = ID3()` is empty (apestrip.py:485-487 and the repair twin at :366-368); every APE field then looks sole-source, and `id3.save(path, v2_version=3, v1=2)` (apestrip.py:515, :403) *rebuilds* ID3v1 from the sparse v2 frames, blanking whatever title/artist/album the old ID3v1 held. Exactly the old rips most likely to carry APE tags. Side effect of the same root cause: the redundancy check never sees v1 values.
  **Fix:** when the `ID3NoHeaderError` fallback fires, seed the fresh `ID3()` from the file's ID3v1 block before planning migrations: read the last 128 bytes, and if they start with `TAG`, parse via `mutagen.id3.ParseID3v1` and `id3.add()` each returned frame. Do this in both the normal path and the repair path (factor a small `_id3_for(path)` helper used by both). The redundancy check then sees the v1 values too, and `save(v1=2)` regenerates ID3v1 faithfully.
  **Test:** fixture MP3 with ID3v1 (title/artist set) + APEv2 carrying one TXXX-bound field, run `--keep-metadata`, assert ID3v1 title/artist survive and the APE field migrated.

- [ ] **M2 (apestrip.py): multi-value APE text items flattened to a NUL-embedded string; values after the first are effectively lost.** `_ape_text` (apestrip.py:160-167) does `str(value)`; mutagen's `APETextValue.__str__` joins values with `\x00` (the raw-repair `_RawAPEValue.__str__` at :301-302 preserves NULs the same way). Saved as v2.3 UTF-16, the embedded NUL terminates the string, so foobar2000/DeaDBeeF show only the first value. Under a flag whose contract is "nothing is lost".
  **Fix:** have `_ape_text` return `str(value).split("\x00")` (a list; single-value items become a one-element list), and pass that list straight to `text=` in `apply_migration` for both native frames and TXXX (mutagen text frames accept lists). Update `migration_label` to render lists readably (`" / ".join`).
  **Test:** APE `Artist` with two values migrates to a TIT-family frame whose `.text` has both entries.

- [ ] **M3 (apestrip.py): repair path drops file permissions to 0600.** `mkstemp` (apestrip.py:396) creates the temp file mode 0600 and `os.replace` (apestrip.py:409) installs it verbatim; every `--repair-malformed` repair silently strips group/other access, which breaks shared-mount libraries.
  **Fix:** before `os.replace`, copy the original's mode: `os.chmod(tmp, os.stat(path).st_mode & 0o7777)` (or `shutil.copymode(path, tmp)`).
  **Test:** chmod a fixture to 0644, repair, assert `stat.S_IMODE` still 0644.

- [ ] **M4 (apestrip.py): repair-path I/O errors abort the whole run.** `open(path, "rb")` (apestrip.py:354) and `mkstemp` (apestrip.py:396) sit outside any try; one unreadable file or read-only directory raises `OSError` up through `_handle_malformed` → `process_file` → the unguarded loop in `main` (apestrip.py:658), killing a library-wide run mid-write-pass. The normal path wraps its mutations (:507-523); the repair path is asymmetric.
  **Fix:** wrap the raw-read/repair body in `try/except OSError as e: return FileResult(path, had_ape=True, error=str(e))` so the loop's existing error accounting handles it.
  **Test:** chmod-0 fixture inside a two-file run; assert the run completes and the summary counts one error.

- [ ] **M5 (apestrip.py): repair excision treats everything between the APE footer and a strictly-terminal ID3v1 as junk.** `_parse_raw_ape` (apestrip.py:324, :342, :394) computes `id3v1_start = len(data) - 128` only when the *last* 128 bytes start with `TAG`, else `len(data)`, and excises `data[header_start:excise_end]`. Two traced losses (repair mode only): a Lyrics3 block sitting between APE and ID3v1 is deleted as junk; an ID3v1 followed by any trailing byte (broken downloader padding) is no longer end-anchored, so the excision deletes the ID3v1 itself. The structural proof passes and verification passes in both cases.
  **Fix:** tighten the excision to "the APE block exactly, plus only *recognized* trailing structures". After locating the APE block, examine the bytes between its end and EOF: allow (and preserve) a Lyrics3v2 block (`LYRICSBEGIN`...`LYRICS200`) and an ID3v1 (`TAG` at 128 from the end of recognized content); if any unrecognized bytes remain between the APE end and the recognized trailer chain, refuse the repair and report `unrecognized trailing data (not repaired)` instead of cutting. The excision then removes `[header_start, ape_end)` only, leaving trailers byte-for-byte.
  **Tests:** `audio|APE|Lyrics3|ID3v1` fixture repairs with Lyrics3 intact; `audio|APE|ID3v1|1 junk byte` is refused (reported, file untouched); the existing clean layouts still repair.

- [ ] **M6 (apestrip.py): log asserts "stripped APEv2 tag" for files it did not touch.** The execution loop's else-branch (apestrip.py:679-682) prints the stripped line without checking `r.stripped`; a file whose APE tag vanished between the planning and write passes gets a false audit-log entry.
